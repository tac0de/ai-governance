AI Governance Architecture v0.1
(Deterministic Trace 기반 중앙 거버넌스 체계)
0. 목적

이 아키텍처는 LLM/Agent 시스템을 지능 중심이 아니라 통제·재현성·비용 최적화 중심으로 설계한다.

핵심 목표:

재현 가능한 의사결정 구조

토큰 비용 최소화

증거 기반 판단

감사 가능한 Trace

실행 수단과의 분리 (엔진 비종속)

1. 철학적 전제 (Non-Negotiable Principles)
P1. 중앙은 실행자가 아니라 판정자다.

중앙 레이어는 직접 문제를 풀지 않는다.
중앙은 다음만 수행한다:

의도 확인

작업 분해 승인

증거 요구

결과 판정

재시도/에스컬레이션 결정

P2. 상태는 설명하지 않는다. 참조한다.

모든 상태는 해시 기반 참조로 관리한다.

STATE_REF = sha256(serialized_state)
EVIDENCE_REF = sha256(evidence_blob)
PLAN_REF = sha256(plan_structure)

LLM에게 상태를 반복 서술하지 않는다.

P3. 모든 행동은 Opcode로 고정한다.

자연어 명령 금지.
행동은 사전 정의된 opcode 집합만 사용.

P4. 모든 판단은 Evidence 기반이다.

중앙은 “모델의 주장”이 아니라 “증거 묶음”으로 판정한다.

중앙은 결정을 기록만 한다.

4. Envelope 명세

모든 메시지는 다음 구조를 따른다.

{
  "intent_id": "uuid",
  "policy_profile": "strict|balanced|creative",
  "state_ref": "hash",
  "plan_ref": "hash|null",
  "budget": {
    "max_tokens": 4000,
    "max_steps": 8
  },
  "acceptance_tests_ref": "hash|null"
}
5. Opcode 집합 (v0.1 Minimal Set)
Opcode	설명
PLAN	작업 분해
DECOMPOSE	세부 태스크 분리
CALL_TOOL	외부 호출
COLLECT_EVIDENCE	자료 수집
SUMMARIZE_EVIDENCE	요약
EVAL	결과 평가
MERGE	결과 병합
RETRY	재시도
ESCALATE	상위 판단 요청
FREEZE_STATE	상태 고정

확장 금지. 12개 이하 유지.

6. Trace Record 구조

모든 단계는 불변 로그로 저장.

{
  "prev_hash": "hash",
  "opcode": "CALL_TOOL",
  "args_hash": "hash",
  "output_hash": "hash",
  "evidence_refs": ["hash1","hash2"],
  "cost": {
    "tokens_used": 812
  },
  "verdict": "approved|rejected|retry"
}

Trace는 append-only.

7. Deterministic Trace 규칙

완전 결정론을 요구하지 않는다.
다음만 고정한다:

분기 조건

승인 기준

재시도 조건

예산 초과 처리 규칙

LLM 출력 문장은 비결정적일 수 있으나,
판정 구조는 결정적이어야 한다.

8. MCP Adapter 설계 원칙

MCP는 “도구 연결 규격”이다.
우리 시스템에서는 Evidence 공급 계층으로 정의한다.

Evidence 스키마
{
  "source": "jira|github|web|db",
  "timestamp": "iso8601",
  "query": "string",
  "content_ref": "hash",
  "confidence": 0.92,
  "access_level": "internal|public"
}

MCP Adapter는:

정규화

중복 제거

요약

캐시

권한 필터링

을 수행한다.

중앙은 MCP를 직접 호출하지 않는다.
CALL_TOOL → MCP Adapter → Evidence 반환 구조.

9. 비용 최적화 규칙

상태는 항상 ref 전달

Evidence는 전문이 아니라 요약 + ref

동일 plan 재사용 시 plan_ref 재활용

실패 시 diff만 재전송

policy_profile을 반복 프롬프트로 쓰지 말고 ID로 참조

10. Acceptance Test 체계

모든 Intent는 Acceptance Test를 가진다.

예:

{
  "must_include_sources": true,
  "max_hallucination_score": 0.1,
  "required_sections": ["summary","analysis","risk"]
}

EVAL opcode는 이 기준에 따라 자동 판정.

11. 중앙 거버넌스의 철학적 정의

이 아키텍처는 AGI를 지향하지 않는다.

지향점은:

자기 통제

재현 가능성

책임 구조

증거 기반 의사결정

확장 가능한 질서

지능이 아니라 질서를 설계한다.

12. 구현 우선순위

Trace Engine (append-only)

Envelope Validator

Opcode Dispatcher

Evidence Store

MCP Adapter 최소 구현

Policy Profile 관리

Acceptance Evaluator

13. 성공 기준

동일 intent + 동일 evidence → 동일 판정

평균 토큰 사용량 지속 감소

재시도율 감소

Trace 기반 감사 가능
